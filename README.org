#+TITLE: org-agent
#+SUBTITLE: Orchestrate AI agents via org-mode headings
#+STARTUP: overview

* What is this?

You have a project.  You open =TODO.org=.  Each heading is a task.
You press =C-c C-x s= on a heading and an AI agent spawns — isolated
in its own git worktree, scoped to that task, with full context from
the org tree.  When it finishes, it marks itself done.  Its changes
auto-commit on a branch.  The parent heading merges everything.

That's org-agent.  Org-mode headings as agent sessions.  Git worktrees
as sandboxes.  The tree is the persistent state — sessions come and go,
headings survive.

Requires [[https://github.com/stevemolitor/claude-code.el][claude-code.el]] for terminal management.

* Install

#+begin_src elisp
;; Add to load-path (adjust path to where you cloned this repo)
(add-to-list 'load-path "~/repos/org-agent")

(use-package org-agent
  :after (org claude-code)
  :config
  (org-agent-mode 1))
#+end_src

* Tutorial: your first agent in 5 minutes

This walks you through the full cycle: create a task, spawn an agent,
watch it work, collect the result.

** Step 0: initialize your project

Open a project in Emacs, then:

: M-x org-agent-init-project

This creates:
- =.org-agent/= directory for state
- Hooks in =.claude/settings.json=
- A starter =TODO.org= if you don't have one
- Template =.claude/CLAUDE.md= and =.agents.md=

** Step 1: create a task

Open =TODO.org=.  Add a heading with a bracket ID:

#+begin_example
,* NEXT [A-1] Add a hello-world endpoint
#+end_example

The bracket ID (=[A-1]=) is how everything references this task —
agents, messages, branches, worktrees.

NEXT means "ready to execute."  The agent gets full edit access
and an isolated git worktree.

** Step 2: spawn the agent

With point on the heading, press =C-c C-x s=.

What happens:
1. A git worktree is created at =.org-agent/worktrees/A-1/=
2. A branch named =A-1= forks from your current branch
3. A Claude Code terminal opens with full task context
4. The agent starts working in its isolated copy of the repo

You can keep using your editor normally — the agent works in a
separate directory.

** Step 3: watch it work (or don't)

The agent runs in a terminal buffer.  You can:
- Switch to it: =C-c C-x o= (switch session)
- Check status: =C-c C-x l= (dashboard)
- Send it a message: =C-c C-x s= again on the same heading
- Ignore it and do other work

** Step 4: agent finishes

The agent marks itself done:

#+begin_src bash :eval no
emacsclient -e '(org-agent-mark-done "A-1")'
#+end_src

What happens automatically:
1. All changes in the worktree are committed on branch =A-1=
2. The =:OWNS_PATHS:= property is set from the git diff
3. The session closes
4. You get notified

** Step 5: review the diff

Before merging, review what the agent did.  With point on the DONE
heading, press =C-c C-x g=.  This opens a magit diff showing
everything the agent changed (falls back to a plain diff buffer if
magit isn't installed).

** Step 6: merge the result

Set the parent heading (or the heading itself, if it's top-level) to
DONE.  The auto-merge hook merges branch =A-1= into your working
branch and cleans up the worktree.

That's it.  One heading, one agent, one branch, auto-merged.

* Tutorial: decomposing a big task

Real work isn't one heading.  Here's how to break a task into
parallel agents.

** Step 1: create a parent task

#+begin_example
,* TODO [B-1] Add user authentication
#+end_example

TODO (not NEXT) means "needs decomposition."  This agent gets
read-only access — it plans, it doesn't code.

** Step 2: spawn a planning agent

Press =C-c C-x p= on the heading.  The planning agent reads your
codebase and suggests a decomposition.

Or create children yourself:

#+begin_src bash :eval no
emacsclient -e '(org-agent-add-child "B-1" "[B-1a] JWT token service" "NEXT")'
emacsclient -e '(org-agent-add-child "B-1" "[B-1b] Login endpoint" "NEXT")'
emacsclient -e '(org-agent-add-child "B-1" "[B-1c] Auth middleware" "NEXT")'
#+end_src

** Step 3: dispatch all children

#+begin_src bash :eval no
emacsclient -e '(org-agent-dispatch-batch (list "B-1a" "B-1b" "B-1c"))'
#+end_src

Or press =C-c C-x b= on the parent heading to batch-spawn all
NEXT leaves.

Three agents start simultaneously, each in its own worktree on its
own branch.  They can't interfere with each other.

** Step 4: children finish

As each child marks done, you get notified:

: Child [B-1a] (JWT token service) is DONE. Progress: 1/3 children DONE.

** Step 5: merge everything

When all children are DONE, set =[B-1]= to DONE.  The hook merges
all three branches, resolves any conflicts (or stops and tells you),
and cleans up worktrees.

* Tutorial: sequential chains

Sometimes child B needs files that child A creates.  Use a chain:

#+begin_src bash :eval no
emacsclient -e '(org-agent-add-chain "C-1"
  (list "[C-1a] Define database schema"
        "[C-1b] Implement repositories"
        "[C-1c] Add integration tests"))'
#+end_src

- =C-1a= starts immediately as NEXT
- =C-1b= is blocked until =C-1a= completes — then auto-dispatches
  with =C-1a='s work already in its worktree
- =C-1c= waits for =C-1b=, same pattern

Use chains only when there's a real file dependency.  Default to
parallel.

* Messaging between agents

Agents can talk to each other via emacsclient.

#+begin_src bash :eval no
# Send a live message to an agent's terminal (immediate)
emacsclient -e '(org-agent-send "A-1" "Focus on error handling first." t)'

# Add a persistent note to a heading (survives restarts)
emacsclient -e '(org-agent-add-note "A-1" "Found: need to handle token expiry.")'

# Read another agent's subtree (inspect state)
emacsclient -e '(org-agent-read-subtree "A-1")'

# Broadcast to all live agents
emacsclient -e '(org-agent-broadcast "Pause and report status.")'
#+end_src

Always pass =t= as the third argument to =org-agent-send= — it
bypasses the queue and delivers immediately.

* Keybindings

All under =C-c C-x= in org-agent-mode:

| Key | What it does                         |
|-----+--------------------------------------|
| s   | Spawn agent on heading at point      |
| b   | Batch-spawn all NEXT leaves          |
| p   | Spawn planning agent (decompose)     |
| r   | Spawn review agent for DONE children |
| q   | Close review/read-only sessions      |
| k   | Kill session (C-u to also purge)     |
| d   | Purge session from disk (fresh start)|
| g   | Show git diff for this node          |
| l   | Open agent dashboard                 |
| o   | Switch to another agent session      |
| n   | Capture a TODO from any buffer       |
| j   | Jump to next waiting agent           |

Global (when =org-agent-global-keybindings= is non-nil):

| Key   | What it does                    |
|-------+---------------------------------|
| C-c n | Capture a TODO from any buffer  |
| C-c j | Jump to next waiting agent      |

* Configuration

** Paths

#+begin_src elisp :eval no
(setq org-agent-org-filename "TODO.org")             ;; task file name
(setq org-agent-state-directory ".org-agent")         ;; state/sessions dir
(setq org-agent-worktree-directory ".org-agent/worktrees")  ;; worktree dir
(setq org-agent-instructions-file ".claude/CLAUDE.md")      ;; project instructions
#+end_src

** Pre-commit formatting

#+begin_src elisp :eval no
;; Rust
(setq org-agent-pre-commit-formatter "cargo fmt"
      org-agent-merge-formatter "rustfmt")

;; Python
(setq org-agent-pre-commit-formatter "black .")

;; JavaScript
(setq org-agent-pre-commit-formatter "prettier --write .")

;; None (default)
(setq org-agent-pre-commit-formatter nil)
#+end_src

** Methodology injection

Every agent gets methodology text in its prompt.  Override it:

#+begin_src elisp :eval no
;; Point to a file
(setq org-agent-methodology-file "docs/methodology.md")

;; Or set inline
(setq org-agent-methodology-text "## Our process\n1. Write tests first\n...")

;; Or use the default TDD stub (built-in)
#+end_src

** Backend selection

#+begin_src elisp :eval no
;; Global default
(setq org-agent-default-backend "anthropic")  ;; "anthropic", "codex", "zai"

;; Per-heading (inheritable via org properties)
(org-agent-set-backend "A-1" "codex")
#+end_src

* Hooks

Elisp hooks for extending org-agent:

| Hook                          | When it fires                    |
|-------------------------------+----------------------------------|
| org-agent-pre-execute-hook    | Before spawning (receives plist) |
| org-agent-post-execute-hook   | After agent completion           |
| org-agent-state-change-hook   | On TODO state change             |

Claude Code PreToolUse hooks (shell scripts) enforce workflow rules:

- *Edit guard*: blocks edits when agent state isn't NEXT/TODO, or
  when writing outside declared =:OWNS_PATHS:=
- *Commit guard*: blocks =git commit= for NEXT agents without
  worktrees (forces the auto-commit path)

Both are installed by =org-agent-install-hooks= / =org-agent-init-project=.

* Adding a new backend

#+begin_src elisp :eval no
(push '("my-backend" . (:require my-backend-code
                         :term-make my-backend--term-make
                         :term-send-string my-backend--term-send-string
                         :term-configure my-backend--term-configure
                         :term-setup-keymap my-backend--term-setup-keymap
                         :term-customize-faces my-backend--term-customize-faces
                         :event-hook my-backend-event-hook))
      org-agent-backend-adapters)
#+end_src

* Migrating from claude-org

All =claude-org-*= functions are aliased to =org-agent-*=.  Existing
guard scripts and running agents continue to work.

Point your paths at the old locations:

#+begin_src elisp :eval no
(setq org-agent-state-directory ".claude/tree"
      org-agent-worktree-directory ".claude/worktrees"
      org-agent-instructions-file ".claude/CLAUDE.md"
      org-agent-pre-commit-formatter "cargo fmt")
#+end_src
